# Emergent Competencies in Locally-Executed Algorithms

An autonomous overnight research experiment investigating whether the emergent properties discovered in Levin et al.'s sorting algorithm research generalize to other computational domains.

## Background

[Zhang, Goldstein & Levin (2025)](https://arxiv.org/abs/2401.05375) showed that sorting algorithms, when re-framed from top-down execution to autonomous "cell-view" execution (each element runs its sorting policy locally), exhibit surprising emergent behaviors: error tolerance, detour navigation around defects, and spontaneous clustering in mixed-algorithm arrays. We asked: **do these phenomena generalize beyond sorting?**

## Setup

The project was bootstrapped by cloning [Karpathy's autoresearch](https://github.com/karpathy/autoresearch) and removing its `.git` directory to avoid a nested git repo. The original autoresearch code (designed for GPU-based neural network optimization) is preserved in `autoresearch/` as reference. All new experiment infrastructure was built on top.

The experiment was run entirely autonomously overnight on a local machine:

- **Research agent**: Gemma 4 26B A4B IT (Q4_K_M) via LM Studio
- **Agent harness**: Custom Python script (`agent.py`) calling the OpenAI-compatible API
- **Orchestration**: Claude Code (Opus 4.6) monitoring, intervening when the agent got stuck
- **Hardware**: Apple M4 Pro (no GPU required - all experiments are CPU-only Python + NumPy)

## Key Findings

### 1. Interaction Range Predicts Robustness

Sorting algorithms tested under increasing rates of "frozen" (damaged) cells:

| Algorithm | Interaction | 0% dmg | 10% dmg | 20% dmg | 30% dmg |
|-----------|------------|--------|---------|---------|---------|
| Selection | Non-local | 100% | 100% | 80% | 27% |
| Bubble | Bidirectional local | 100% | 97% | 57% | 3% |
| Gnome | Sequential local | 100% | 15% | 5% | 0% |
| Insertion | Unidirectional local | 100% | 0% | 0% | 0% |

Algorithms that interact with distant elements bypass damage; local algorithms get blockaded.

### 2. Damage Can Be Constructive

In distributed consensus, frozen nodes act as **anchors** that *increase* agreement:

| Damage Rate | Agreement |
|------------|-----------|
| 0% | 0.498 |
| 20% | 0.678 |
| 40% | 0.824 |
| 60% | 0.926 |

The opposite of sorting. Structured damage helps; random damage hurts.

### 3. Stochasticity Enables Robustness

- Random walk (20% success) beats deterministic gradient search (5%) in damaged environments
- Sensor dropout *improves* k-means clustering from 12% to 100% (analogous to dropout regularization)
- Optimal noise exists: k-means success peaks at noise=2.0 (67%), drops at noise=4.0 (50%)

### 4. Failure-Induced Detection Paradox

In anomaly detection at high noise, sensor failures actually *improve* detection (43% -> 63%). Dead sensors reduce background variance, making anomalies more visible.

### 5. Taxonomy of Damage Effects

| Factor | Destructive | Constructive |
|--------|------------|--------------|
| Alignment | Random damage (sorting) | Structured/consistent (consensus anchors) |
| Interaction | Local-only algorithms | Non-local algorithms bypass damage |
| Stochasticity | Deterministic agents get trapped | Random agents explore around obstacles |
| Redundancy | Single-shot execution | Multiple retries overcome noise |

## How It Works

### The Agent Loop

1. The Gemma model reads `program.md` for research instructions
2. It creates experiment files in `experiments/`, runs them, analyzes results
3. It communicates via XML action tags (`<action type="shell">`, `<action type="write_file">`, etc.)
4. Results are parsed from stdout and fed back as context
5. The agent decides what to try next based on findings

### Adaptations from Karpathy's autoresearch

The [original autoresearch](https://github.com/karpathy/autoresearch) framework is designed for GPU-based neural network optimization. We adapted it for CPU-only algorithmic experiments:

- Replaced `train.py` (GPT model) with `experiment.py` (sorting/consensus/clustering algorithms)
- Replaced `val_bpb` metric with domain-specific metrics (success rate, disorder, agreement)
- Built `agent.py` as a lightweight LLM-driven harness for LM Studio
- Added anti-stuck detection (redirects the agent when it loops)
- Added file-write safety checks (prevents truncated overwrites)

### Monitoring

Claude Code ran a cron job every 7 minutes checking:
- Is the agent process alive?
- Is the log growing?
- Is it stuck in a repetition loop?

Intervened 3 times overnight to redirect the agent when it got stuck summarizing instead of experimenting.
